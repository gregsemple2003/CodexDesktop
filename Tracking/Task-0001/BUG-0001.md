# Bug 0001

## Summary

The live app does not respond to the required `Ctrl+Alt+Space` global hotkey on this Windows machine.

## Current Evidence

- regression run `0001` already showed that the automated smoke harness could not trigger the registered hotkey path
- on `2026-04-02`, the human launched `python -m app.codex_dashboard` from `C:\Agent\CodexDashboard`
- the app opened a console window
- pressing `Ctrl+Alt+Space` did nothing
- the overlay did not appear

## Why This Matters

The task acceptance criteria explicitly require a hotkey-first overlay that can be shown and hidden while the app stays backgrounded.

The current build therefore fails task-level regression and is not closure-ready.

## Current Hypothesis

The hotkey registration or `WM_HOTKEY` delivery path is broken in the real Tk event loop, not only in the synthetic smoke harness.

Likely areas:

- Win32 `RegisterHotKey` setup
- the message-pump polling path in `hotkey.py`
- the Tk scheduling loop that is supposed to drain the hotkey messages

## Next Step

Debug the real hotkey path locally, then rerun regression from the live app surface:

1. inspect registration success and the live `WM_HOTKEY` polling path
2. implement a fix
3. rerun unit tests
4. rerun the desktop-app regression lane and confirm `Ctrl+Alt+Space` toggles the overlay
