# Bug 0001

## Summary

Resolved: the live app now responds to `Ctrl+Alt+Space` again after moving global-hotkey ownership onto a dedicated Win32 message-loop thread.

## Failure Evidence

- regression run `0001` showed that the automated smoke harness did not trigger the registered hotkey path
- regression run `0002` then confirmed the product bug on the live app surface:
  - the human launched `python -m app.codex_dashboard`
  - the process stayed open
  - pressing `Ctrl+Alt+Space` did nothing
  - the overlay did not appear

## Root Cause

The original implementation registered the hotkey on the Tk main thread and then tried to poll `WM_HOTKEY` messages with `PeekMessageW`.

That design competed with Tk's own Windows message loop for the same thread queue, so the app could miss the hotkey message instead of reliably toggling the overlay.

## Resolution

The fix changed `app/codex_dashboard/hotkey.py` to:

- register the hotkey on a dedicated daemon thread
- own a blocking `GetMessageW` loop on that thread
- queue pending callbacks back to the main app for safe polling and UI toggling
- cleanly post `WM_QUIT` and unregister the hotkey on shutdown

## Resolution Evidence

- `python -m unittest discover -s tests -p "test_*.py" -v`
  - `8` tests passed
- regression run `0003`
  - `python -m app.codex_dashboard --smoke-artifact-dir Tracking/Task-0001/Testing/REGRESSION-UI-SMOKE-0002`
  - `overlay-summary.txt` recorded:
    - `hotkey_triggered=True`
    - `overlay_fallback=False`

## Remaining Limits

- the passing rerun uses a synthetic system keyboard event rather than a fresh human physical-keyboard spot check after the fix
- startup-folder behavior across a real Windows logon remains outside this task
