# BUG-0010: Open Task Buttons Do Not Raise Docs In VSCodium

Status: closed

## Summary

`Open Task` actions on the `Tasks` tab are intended to raise the relevant task
docs in VSCodium, but the human reports that this is not happening.

## Visible Evidence

Human report on 2026-04-27:

- selected task: `Task-0011`
- visible actions include multiple `Open Task` buttons
- intended effect: raise task docs in VSCodium
- actual observed effect: no VSCodium doc opens or raises

## Why This Is A Product Bug

The `Tasks` tab is supposed to be a direct context-recovery surface. If an open
action does not raise the relevant docs in VSCodium, the human is pushed back
into manually hunting through task artifacts.

## Likely Root Cause

Backend launch targets may provide a `code` command while the human's editor
lane expects VSCodium, or the frontend may not gracefully fall back from an
unavailable command to a concrete file/folder URI. The UI also does not make the
target identity clear enough when several task artifacts are present.

## Expected Behavior

- Task open actions raise the intended task artifact or folder in VSCodium when
  an editor launch command is available.
- If the provided command is unavailable, the dashboard falls back to a concrete
  normalized file/folder path rather than silently doing nothing.
- The visible action label identifies what will open, such as `Open Task`,
  `Open Plan`, or `Open Handoff`.

## Required Closure Proof

- Unit coverage proves task launch targets normalize `file:///C:/...` URIs to
  local Windows paths before fallback launch.
- Unit coverage proves unavailable allowed editor commands fall back to the URI
  path instead of failing before a launch attempt.
- Actual dashboard or smoke evidence shows distinct task open actions for a
  planning task with `TASK.md`, `PLAN.md`, and `HANDOFF.md`.

## 2026-04-27 Fix

Status changed to closed.

Implementation:

- the launch path now resolves backend-provided `code` commands to installed
  VSCodium-compatible command fallbacks such as `codium`
- if no allowed editor command is available, the launch path falls back to the
  normalized file URI instead of failing before any concrete target is tried
- action labels now tell the human which artifact will open

Validation:

- `test_open_task_launch_target_falls_back_from_code_to_vscodium` proves an
  unavailable `code` command resolves to `codium`
- `test_open_task_launch_target_falls_back_to_normalized_file_uri` proves
  `file:///C:/...` is normalized to a local Windows path before fallback launch
- app-surface smoke evidence:
  [overlay-summary.txt](./Testing/BUG-0007-20260427-121303/overlay-summary.txt)
  records `tasks_selected_actions=Dispatch,Open Task,Open Handoff,Open Plan`
