# BUG-0006: No-Run Tasks Expose Run-Control Actions Without Visible Reasons

Status: closed

## Summary

The selected `Task-0001` detail exposes `Pause` and `Poke` action labels even
though backend readback says no active run exists.

The buttons are disabled in the widget state, but the visible detail surface
does not show the backend block reason beside those controls. REG-004 requires
no-active-run controls to be absent or disabled with a true reason.

Regression artifact:

- [REG-004-20260427-0048.md](./Testing/REG-004-20260427-0048.md)

Supporting artifacts:

## Durable/Backend Evidence

Backend `/api/v1/tasks` for `Task-0001` returned:

```json
{
  "current_story": {
    "status": "no_active_run",
    "reason": "No task run currently owns the live story."
  },
  "state_envelope": {
    "action_block_reasons": {
      "interrupt": [
        {
          "code": "no_active_run",
          "summary": "Interrupt is unavailable until a task run exists."
        }
      ],
      "poke": [
        {
          "code": "no_active_run",
          "summary": "Poke is unavailable until a task run exists."
        }
      ]
    }
  }
}
```

## Why The Visible Claim Is False Or Unsupported

A visible `Pause` or `Poke` control implies there is a run to pause or poke.
For a no-active-run task, the surface must either omit those controls or show
the true block reason where the human can see it.

Disabled buttons alone do not reconcile the visible action availability claim
against backend truth.

## Likely Root Cause

The frontend action renderer creates a button for every backend action
availability object and stores the block reason in the action payload, but the
detail pane does not render that reason next to disabled buttons.

## Required Closure Proof

- For no-active-run tasks, `Pause` and `Poke` are absent, or disabled with
  visible reason text that matches backend block reasons.
- The visible stop/hold copy remains `Pause`, not backend-internal
  `Interrupt`.
- REG-004 captures the selected detail action area and reconciles action
  visibility against backend `actions` and `action_block_reasons`.
