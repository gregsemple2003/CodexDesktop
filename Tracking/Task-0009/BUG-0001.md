# BUG-0001: Service Lane Restart Rebuilt From Mutable Checkout

Status: fixed and promoted to human service lane with dirty-source manifest

## Summary

The service-lane scheduled task could restart through
`backend/orchestration/scripts/Run-OrchestrationLane.ps1` in the mutable repo
checkout. That runner rebuilt `controlplane-service-lane.exe` from the current
checkout on every restart, so repo updates or agent edits could become a
delayed human-lane deployment.

## Impact

- Local repo mutation could reach the human lane without an explicit promotion.
- A scheduled-task restart could deploy code different from the last intended
  human-lane state.
- Health checks and successful command exits did not prove the human lane was
  preserved, pinned, or running intended code.

## Root Cause

The human lane was coupled to the mutable development checkout:

- scheduled task action pointed at a repo-local launcher
- launcher built from the repo at start time
- launcher ran the binary from a mutable lane `bin` path
- status proof did not include pinned release identity, binary hash, launcher
  path, or process path

## Fix

Implemented a release-pinned service-lane path:

- `Publish-ServiceLaneRelease.ps1` builds a release under the service-lane
  runtime root and writes `current-release.json`
- the release manifest records git commit, dirty-source status, binary path,
  binary hash, compose-file path, and compose-file hash
- the scheduled-task launcher is copied under the service-lane runtime root
- `Run-OrchestrationLane.ps1` uses the pinned manifest for the service lane and
  does not build from the repo for service-lane restarts
- `Start-ServiceLane.ps1` and `Install-ServiceLane.ps1` fail closed without a
  valid pinned release and runtime-root launcher
- validation lanes still build from the checkout because they are disposable

## Required Proof

Do not claim this bug is closed for the human lane unless evidence shows:

- `Test-ServiceLaneIsolation.ps1` reports a current release and
  `scheduled_task_uses_pinned_launcher = true`
- `Get-ServiceLaneStatus.ps1` reports a healthy backend
- the running service process path equals the manifest `binary_path`
- the manifest hashes validate
- no repo-local service runner process is supervising the human lane

## Evidence

- `python -m unittest tests.test_service_lane_scripts -v` passed before
  promotion.
- `python -m unittest discover -s tests -p "test_*.py" -v` passed before
  promotion.
- PowerShell parse check passed for lane release scripts before promotion.
