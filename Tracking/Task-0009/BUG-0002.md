# BUG-0002: Frontend Was Mutable While Human-Lane Safety Was Claimed

Status: open

## Summary

The backend service lane was release-pinned, but the desktop dashboard frontend
still launched directly from the mutable repo checkout through
`CodexDashboard.cmd`.

I then treated backend-only release proof as if it proved the human-facing
dashboard surface was current and safe. That claim had no epistemic warrant.

## Impact

- The visible dashboard UI could remain an old in-memory process after repo
  updates.
- Restarting the dashboard UI could load arbitrary current repo state rather
  than an explicitly promoted release.
- The user could see no `Tasks` tab even while backend proof looked healthy.
- Human-lane safety/currentness reports were misleading because they omitted
  frontend process, launcher, and visible-surface proof.

## Evidence

Observed state:

- running dashboard process:
  `pythonw.exe -m app.codex_dashboard`
- startup command:
  `cd /d "C:\Agent\CodexDashboard"`
- startup command then launched:
  `"C:\Program Files\Python313\pythonw.exe" -m app.codex_dashboard`
- running frontend process start time:
  `2026-04-26T17:52:20-04:00`
- `Tasks` tab code landed later, beginning with commit time:
  `2026-04-26T21:52:48-04:00`

Therefore, the user-visible dashboard process predated the Tasks-tab code.

## Root Cause

The release-pinning invariant was only applied to the backend service lane. The
desktop frontend remained coupled to the mutable repo checkout and had no:

- release manifest
- hash validation
- runtime-root launcher
- process-path proof
- frontend-specific status script
- visible-surface proof requirement

## Correct Rule

Do not claim the human lane is current or safe unless the claim is proven for
each human-lane component involved:

- backend service lane
- desktop dashboard frontend
- job specs and runtime schedules
- visible product surface, when the claim is about what the user can see

Backend health is not frontend proof. A running frontend process is not
currentness proof unless its release identity is known.

## Required Fix

Add dashboard release pinning equivalent to the backend release model:

- publish dashboard source into a runtime-root release directory
- write `dashboard-current-release.json`
- hash the release contents
- launch the dashboard through a runtime-root launcher
- update startup registration to point at that launcher
- add a status/proof script that reports release id, commit, dirty flag, process
  path, launcher path, and hash validation

## Required Closure Evidence

This bug is not closed until:

- dashboard release publish script exists and is tested
- dashboard status/proof script exists and is tested
- startup command no longer launches from `C:\Agent\CodexDashboard`
- running dashboard process uses the pinned release source root
- `Tasks` tab code is present in the pinned release
- backend `/api/v1/tasks` readback works for the running dashboard backend
- durable data/testing/regression docs describe frontend pinning
