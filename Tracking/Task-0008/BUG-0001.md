# BUG-0001: Service Lane Broke After PowerShell Upgrade

## Status

Fixed locally on 2026-04-26.

## Summary

The Task-0008 aftermath exposed a regression in the always-on orchestration service lane: the `CodexDashboard-Orchestration-ServiceLane` Windows Scheduled Task captured a versioned Store PowerShell executable path.

The task action pointed at:

```text
C:\Program Files\WindowsApps\Microsoft.PowerShell_7.6.0.0_x64__8wekyb3d8bbwe\pwsh.exe
```

After PowerShell upgraded to `7.6.1`, that path no longer existed. The scheduled task could still be listed, but it could not launch the service lane.

## Human Impact

This caused the three backend-controlled daily digest jobs to miss scheduled email delivery after the service lane went down. The human-visible failure was no digest email batch for the last couple of nights.

This is explicitly a burden-reduction failure: a routine PowerShell upgrade should not require the human to know how the orchestration service lane is bootstrapped, inspect Scheduled Task action paths, decode Windows error codes, or restart/reinstall infrastructure manually.

## Evidence

- `Get-ServiceLaneStatus.ps1` reported:
  - `task_state = Ready`
  - `process_running = false`
  - `last_error = Unable to connect to the remote server`
- `http://127.0.0.1:4318/healthz` refused connections.
- `Get-ScheduledTaskInfo -TaskName CodexDashboard-Orchestration-ServiceLane` reported:
  - `LastRunTime = 2026-04-26 6:12:02 PM`
  - `LastTaskResult = 2147942402`
- `2147942402` is Windows `0x80070002`, meaning file not found.
- The scheduled task's `Execute` path did not exist.
- The current Store PowerShell path was:

```text
C:\Program Files\WindowsApps\Microsoft.PowerShell_7.6.1.0_x64__8wekyb3d8bbwe\pwsh.exe
```

- Backend job run history showed the three digest schedules last ran at `2026-04-24T08:00:00Z`.
- There were no scheduled runs for `2026-04-25T08:00:00Z` or `2026-04-26T08:00:00Z`.

## Fix

The service-lane helper now avoids versioned WindowsApps PowerShell package paths when registering the scheduled task.

The resolver prefers stable executable paths:

- `%ProgramFiles%\PowerShell\7\pwsh.exe`
- `%SystemRoot%\System32\WindowsPowerShell\v1.0\powershell.exe`

Only non-WindowsApps fallback paths are accepted after that.

Changed file:

- [LaneHelpers.ps1](../../backend/orchestration/scripts/LaneHelpers.ps1)
- [test_service_lane_scripts.py](../../tests/test_service_lane_scripts.py)

## Local Repair Performed

Reinstalled and restarted the service lane through the backend-controlled bootstrap path:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\Install-ServiceLane.ps1
```

The service lane came back healthy on:

- jobs backend: `http://127.0.0.1:4318`
- Temporal: `127.0.0.1:7233`

All three tracked digest jobs were visible through `GET /api/v1/jobs` with `status = in_sync`.

## Verification

```powershell

pwsh -NoProfile -ExecutionPolicy Bypass -Command ". '.\scripts\LaneHelpers.ps1'; Get-PowerShellExecutablePath"
```

Both resolved to the stable Windows PowerShell path:

```text
C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe
```

After reinstall, the scheduled task action also points at an existing stable executable.

## Remaining Follow-Up

The missed April 25 and April 26 digest emails were not automatically backfilled by Temporal because the managed catchup window is 12 hours and the April 26 04:00 local run was already older than that when the service lane was restored.

Backfilling should be an explicit human-approved action because it intentionally sends real emails.
