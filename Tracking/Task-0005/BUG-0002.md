# BUG-0002 Job Setup Ambiguity Lets Agents Bypass Temporal With Standalone Windows Tasks

## Summary

The repo direction says Git-tracked job specs plus Temporal are the real scheduling path, but the durable docs were not explicit enough to stop an agent from treating `set up a job on my machine` as permission to create a standalone Windows Scheduled Task outside the backend. That produced real runs and emails that were invisible to the dashboard-backed orchestration model and therefore violated the user's stated intent.

## Failure Evidence

- On `2026-04-19`, the digest emails were sent from the legacy Windows task layer at `05:06` through `05:09 America/Toronto`, while the dashboard and backend still showed the Temporal schedule at `04:00 America/Toronto`.
- The hourly `Codex Hourly Email Probe` produced overnight emails, but it existed only as a Windows Scheduled Task and did not appear in `GET /api/v1/jobs`.
- The live repo semantics already said the dashboard should read backend state instead of local Windows reconciliation, but the operational docs did not promote that into a hard instruction for future agent behavior.

## Observed

- a working local scheduler path was treated as good enough even when it bypassed the backend-controlled product path
- legacy Windows scheduling state and Temporal scheduling state were treated as interchangeable
- dashboard-visible correctness was not used as the acceptance test for `job setup`

## Expected

- when the human asks to set up or change a scheduled job in this repo, the default path should be tracked v1 job specs plus backend reconcile into Temporal
- a job should not count as `set up` unless it is visible through backend readback and the dashboard Jobs tab
- standalone Windows Scheduled Tasks should be reserved for backend bootstrap infrastructure or explicit human requests for the legacy Windows-only path

## Root Cause

- the repo docs described the Temporal direction, but they did not state the operator rule strongly enough
- the presence of still-running legacy Windows digest tasks created a misleading nearby mechanism that appeared to solve the immediate problem
- the agent optimized for a working local outcome instead of the repo's architectural boundary and the user's stated product intent

## Durable Fix

The repo front door and jobs docs now say explicitly:

- human job-setup requests in this repo default to Git specs plus Temporal
- backend and dashboard visibility are required proof
- ad hoc Windows Scheduled Tasks are not an acceptable substitute unless the human explicitly asks for that path
