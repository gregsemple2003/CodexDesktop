## Repro

1. Start the repo-local Temporal and Postgres stack.
2. Start the worker-backed control-plane against the current tracked `.codex` corpus.
3. Allow startup reconcile to create or update the tracked schedules.

## Observed

- the startup path releases overdue schedule work immediately instead of keeping the proof bounded to one intended trigger
- all three tracked digest jobs begin real Codex executions
- the resulting Codex executions fail inside the Windows sandbox before they can complete the skill workflow

## Expected

- a live `PASS-0002` proof can be bounded to one explicitly chosen run
- schedule reconcile does not silently release overdue work during proof setup
- the chosen real Codex execution completes far enough to provide honest end-to-end executor proof

## Current Hypothesis

- the schedule compile or create path currently leaves catchup behavior effectively open, so overdue daily schedules are eligible to fire as soon as the runtime comes up
- the separate executor failure is inside the ChatGPT-backed Codex CLI sandbox on this Windows host, not in the control-plane process

## Next Step

- fence the schedule startup path so a proof launch cannot release all overdue tracked schedules
- retry with a bounded single-run path
- if the bounded retry still hits `CreateProcessAsUserW failed: 1920`, carry that forward as the active PASS-0002 blocker with the new focused evidence

## Resolution Status

This is no longer the active blocker for `Task-0005`.

The bounded retry on `2026-04-07` resolved both parts of the original proof failure:

- startup reconcile now keeps the tracked schedules bounded by a short catchup window instead of releasing all overdue digest jobs at once
- the Windows executor failure cleared when the control-plane switched the live `codex exec` path to `--dangerously-bypass-approvals-and-sandbox`
