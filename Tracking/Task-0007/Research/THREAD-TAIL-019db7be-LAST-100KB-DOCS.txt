# Dreaming Process

`Dream` is an optional packet-local analysis annex for one canonical intervention day packet.

Use it when the human wants deeper solution design work that goes beyond the canonical `HumanInputEvents`, `HumanNeeds`, and `HumanInterventionTime` outputs.

Do not treat `Dream` as part of the minimum canonical day-packet shape.

## Shared Focus

Dream uses [FIRST-PRINCPLES.md](./FIRST-PRINCPLES.md) as shared directional context.

Those principles are:

- `truth`
- `compassion`
- `tolerance`

## Purpose

Use `Dream` to:

- explain why the human had to intervene
- understand the burden from the packet and the source material behind it
- compare possible fixes
- choose the highest-leverage winners
- turn every option into a concrete task-shaped proposal and make the winners explicit

Do not use `Dream` for:

- raw transcript dumping
- long-lived scratch output
- packet-local copies of shared prompt bundles

## Normative Structure

Required local shape:

```text
<DayRoot>/Dream/
  README.md
  APPROACH.md
  BURDEN-ANALYSIS.md
  ORTHOGONAL-SOLUTIONS-MATRIX.md
  SessionExcerpts/
    INDEX.json
  Option-Tasks/
    INDEX.md
    PROBLEM-0001-OPTION-A.md
```

Required meanings:

- `README.md`
  - front door to the local Dream annex
- `APPROACH.md`
  - method, scope, evidence base, and intended outputs
- `BURDEN-ANALYSIS.md`
  - the evidence-backed explanation of why the human had to intervene
  - should account for recurring event-level `need_tag` clusters from `HumanInputEvents`
- `ORTHOGONAL-SOLUTIONS-MATRIX.md`
  - the problem set, options, winners, and rollout order
- `SessionExcerpts/INDEX.json`
  - packet-local source neighborhoods kept because they clarify what really happened
  - each excerpt record must carry explicit `source_event_ids` for the `HumanInputEvents` entries materially represented by that excerpt
- `Option-Tasks/`
  - concrete task-shaped proposals for every matrix option
  - these writeups must follow [TASK-CREATE.md](./TASK-CREATE.md)
  - for burden-reduction proposals, they must preserve burden, causal claim, evidence, mechanism choice, human relief, uncertainty, and falsifier
  - when the packet exposes stable event ids, burden-reduction option tasks must also carry explicit `source_event_ids` so a reader can trace the task back to packet truth without reconstructing the chain from summaries alone

## Source-Following Rule

Dream should not stop at a lossy summary when a cited source document would make the analysis truer.

For hypothesis formation, burden analysis, and prioritization:

- start from the packet artifacts
- follow `HumanInputEvents` to `SOURCE-PACKET.jsonl`
- use recurring event-level `need_tag` clusters as an evidence surface, not as final truth
- follow `SOURCE-SESSIONS.json` to the authoritative transcript surface for each session:
  - `session_file` when `transcript_surface = "session_file"`
  - ordered `part_files` when `transcript_surface = "ordered_parts"`
- follow other packet-cited source documents or source `jsonl` logs when they clarify meaning

When the packet exposes stable event ids:

- preserve them explicitly in Dream artifacts instead of collapsing them into summary language alone
- do not rely on excerpt labels or `need_tag` names as a substitute for exact event provenance when later tasks need to trace burden claims back to packet truth

Do not discover `.parts` directories ad hoc.

Do not treat derived JSON alone as a sufficient understanding of the packet.

For burden analysis, prioritization, and solution design, investigate the relevant session logs themselves when they clarify what really happened.

Prefer the nearest source that increases truth.

## Burden Decomposition Rule

Dream may merge burdens in `BURDEN-ANALYSIS.md` when that makes the day easier to understand.

Dream must not merge burdens in `ORTHOGONAL-SOLUTIONS-MATRIX.md` by default.

Treat a burden driver as independent when any of these differ:

- the failure boundary
- the claimed remedy mechanism
- the enforcement boundary
- the acceptance test
- the falsifier

Shared docs, shared schemas, shared generators, shared prompts, or shared rollout order do not make two burdens the same problem.

If pass 2 merges two or more burden drivers into one downstream problem row, it must state all of these explicitly:

- which burden drivers were merged
- the one mechanism that is claimed to solve them together
- the one enforcement boundary where that mechanism acts
- the one acceptance test that would prove the merged remedy worked
- the one falsifier that would prove the merged remedy wrong or incomplete

If that cannot be stated cleanly, split the burdens into separate problem rows and separate option tasks.

Do not collapse distinct kinds of exported human work into one neat solution if that would hide what the human had to carry.

## Pass Ownership

Dream is a two-pass workflow.

Preferred execution mode:

- one clean Dream worker should carry both passes for the same packet
- that worker should reread raw packet evidence and relevant source documents at each pass boundary
- optional final criticism may come from a separate audit agent, but the main Dream run should stay with one worker

Pass ownership is hard:

- pass 1 owns `APPROACH.md`, `SessionExcerpts/INDEX.json`, and `BURDEN-ANALYSIS.md`
- pass 2 owns `ORTHOGONAL-SOLUTIONS-MATRIX.md`, `Option-Tasks/`, `README.md`, and final conformance

Later work may correct earlier abstractions when the evidence demands it.
If that happens, the correction should be named plainly.

## Workflow

1. Ground in the canonical day packet and the source material behind it.
2. Run pass 1.
   - validate chronology before analysis
   - read packet artifacts, `HumanInputEvents`, source packet files, and named source transcripts
   - investigate the session logs themselves for the events and burdens that matter
   - write `APPROACH.md`
   - write `SessionExcerpts/INDEX.json`
   - write `BURDEN-ANALYSIS.md`
   - preserve explicit `source_event_ids` for each excerpt and each major burden driver
3. Run pass 2.
   - reread the raw packet evidence and relevant source material
   - investigate the session logs themselves for the burdens and candidate remedies that matter
   - map burden drivers to downstream problem rows before writing the matrix
   - reconcile that mapping against the major burden-driver inventory from pass 1 before claiming conformance
   - write `ORTHOGONAL-SOLUTIONS-MATRIX.md`
   - write `Option-Tasks/`
   - write or refresh `README.md`
   - carry exact `source_event_ids` forward into downstream problem rows and burden-reduction option tasks
   - run final conformance
   - if the evidence proves an earlier Dream artifact wrong, patch it and say why
4. Prune the annex.

## Cleanup Rules

In a durable Dream annex, do not keep:

- `Temp/`
- packet-local `Writing-Standards/`
- packet-local prompt bundles once the shared prompt set exists under `Orchestration/Prompts/Dream/`
- superseded exploratory notes whose content now lives in `BURDEN-ANALYSIS.md`, `ORTHOGONAL-SOLUTIONS-MATRIX.md`, `SessionExcerpts/`, or `Option-Tasks/`

## Completion Checks

Before a Dream annex is considered complete, confirm:

1. `HumanInputEvents/INDEX.json` is strictly chronological
2. `SessionExcerpts/INDEX.json` exists, is grounded in packet-cited source material, and each excerpt record carries explicit `source_event_ids`
3. `BURDEN-ANALYSIS.md` is evidence-backed
4. recurring event-level `need_tag` clusters from `HumanInputEvents` were either preserved, merged explicitly, or dropped with explanation
5. each major burden driver carries explicit `source_event_ids` for the `HumanInputEvents` entries that materially support it
6. every major burden driver from `BURDEN-ANALYSIS.md` is accounted for exactly once in the burden-to-problem mapping as:
   - `separate_problem_row`
   - `merged_with_justification`
   - or `dropped_with_explanation`
7. each downstream problem row is either:
   - one burden driver
   - or a documented merge with explicit mechanism, enforcement boundary, acceptance test, and falsifier
8. each `separate_problem_row` mapping has a corresponding on-disk problem row in `ORTHOGONAL-SOLUTIONS-MATRIX.md`
9. each `dropped_with_explanation` mapping is named explicitly in `ORTHOGONAL-SOLUTIONS-MATRIX.md` with the drop reason
10. `ORTHOGONAL-SOLUTIONS-MATRIX.md` resolves winners and rollout order
11. every matrix option link resolves to an on-disk option-task file
12. `Option-Tasks/` contains one task-shaped proposal per matrix option based on actual file existence, not intended coverage
13. each option task follows [TASK-CREATE.md](./TASK-CREATE.md)
14. each burden-reduction proposal task names burden, causal claim, evidence, mechanism, human relief, uncertainty, falsifier, and explicit `source_event_ids`
15. repo-local constraints were taken from the packet repo, not the current workspace repo
16. durable links inside the Dream annex use relative links

# First Principles

Use this file when a workflow needs a stable answer to:

- what matters most
- how to read human input
- what should outrank convenience

We are constructing a system that can perform long-horizon tasks.  When a human has to input to intervene, 
that counts as a failure by default, since broadening the horizon requires answering the question: 
"How could we have prevented the human from ever having to intervene?".  That is the prime motivating
design force behind our agent orchestration system.

To get there, we've identified three governing principles that our agent orchestration system must
follow in order to maximize human utility.

## Core Principles

The governing principles are:

- `truth`
- `compassion`
- `tolerance`

## Truth

`Truth` means:

- reality outranks the model's story
- evidence outranks elegance
- observed human-facing failure outranks convenient internal proof

# Task Create

This file defines the shared workflow for creating or rewriting `Tracking/Task-<id>/TASK.md`.

Use it together with:

- `C:\Users\gregs\.codex\AGENTS.md`
- `C:\Users\gregs\.codex\Orchestration\ORCHESTRATION.md`
- `C:\Users\gregs\.codex\Orchestration\FILE-NAMING.md`
- `C:\Users\gregs\.codex\Orchestration\Exemplars\TASK.md`

## Purpose

The point of task creation is not to name a problem area.

The point is to produce a task writeup that lets a later human or agent answer three questions honestly:

1. What exact outcome is wanted?
2. What exact change is proposed?
3. What would prove the task is done?

If a task still allows multiple materially different implementations, including one the human would reject, the task is not written tightly enough yet.

## Shared Task Writeup Quality Bar

A task is only strong enough to create, recommend, or enqueue when a human can read it cold and tell what success and failure look like without hearing the original conversation.

Use this as the shared quality bar for:

- direct task creation
- rewritten task proposals
- harvested next-task candidates
- burden-reduction proposals

Default to `concrete implementation` when the human is deciding whether to enqueue real work now.

If the evidence only supports `consensus` or `research`, say that explicitly instead of writing a fake implementation task.

Every strong task writeup should make it easy to answer these questions honestly:

- who is affected
- what outcome is wanted in plain English
- what is true now
- why the gap matters
- what solution shape is being proposed
- what concrete files, artifacts, fields, sections, checks, or gates will change
- what would prove the task done
- what does not count as done when weak closure is a real risk

Shared wording rules:

- state the human-perceived or operator-perceived outcome first when the task touches a human-facing surface
- state the technical seam second
- separate clearly:
  - target truth
  - current truth
  - current fallback or proxy truth when that fallback could be mistaken for success
- prefer hard requirements and ordinary language over mushy verbs such as `improve`, `support`, `handle`, `investigate`, or `address` unless the task is honestly research-only
- prefer positive expected-resolution wording over negative-only wording when a person should be able to tell what they will see, understand, or be able to do afterward
- when a degraded fallback or proxy proof could be mistaken for completion, say explicitly that it does not count
- if the evidence supports only a narrow technical seam but not yet the full user-facing resolution wording, stop and write a consensus or research task instead of pretending the task is ready for concrete implementation

## Task Writeup Types

Not every task needs the same level of specificity.

Choose one of these writeup types before writing `TASK.md`.

### 1. Concrete Implementation Task

Use this when the human is deciding whether to enqueue real work now.

This is the default for solution proposals, follow-up fixes, concrete workflow improvements, and other tasks that are meant to become executable backlog items.

Required bar:

- the main solution shape is already chosen
- the task names the concrete files, artifacts, fields, sections, checks, or behaviors that will change
- a later agent should not need to invent the main approach during implementation

### 2. Consensus Task

Use this when the real blocker is an unresolved product, policy, or workflow choice.

Required bar:

- the task names the exact decision that must be made
- the task names the options or tensions that must be resolved
- the output is a decision artifact, not disguised implementation

Do not write a fake implementation task when the real work is still choosing between solution shapes.

### 3. Research Task

Use this when key facts are still missing and those facts could change the design.

Required bar:

- the task names the specific questions that must be answered
- the task names the sources or code areas that must be checked
- the output is a bounded research artifact that should collapse uncertainty

Do not hide open research inside mushy implementation wording like `investigate`, `improve`, or `handle`.

## Base `TASK.md` Sections

Every new or rewritten `TASK.md` must include:

- `## Title`
- `## Summary` or `## Context`
- `## Goals`
- `## Non-Goals`
- `## Implementation Home`
- `## Acceptance Criteria`

Optional sections are allowed when they make the task harder to misread, such as:

- `## Constraints And Baseline`
- `## Proposed Changes`
- `## Expected Resolution`
- `## What Does Not Count`
- `## Open Questions`
- `## Proof Plan`
- `## References`

## Extra Requirements By Writeup Type

### Concrete Implementation Task

In addition to the base sections, concrete implementation tasks should usually include:

- `## Proposed Changes`
- `## Expected Resolution` when the task touches a human-facing surface or workflow
- `## What Does Not Count` when there is a real risk of proxy closure or degraded fallback closure
- `## Proof Plan` when the proof path is easy to fake or misread

Concrete implementation tasks must name:

- the chosen solution shape
- the concrete files, docs, prompts, schemas, generators, or artifacts that will be added or edited
- any exact fields, sections, reply modes, manifests, or gates that must exist afterward
- the exact boundary where the new behavior will be enforced

Concrete implementation tasks must not:

- leave two or three equally live solution shapes inside one task
- say only `add support`, `improve`, `handle`, `address`, or `tighten` without naming the resulting mechanism
- push the main design choice into implementation time

### Burden-Reduction Proposal Tasks

Use this concrete-implementation shape when the task is a proposed fix derived from:

- intervention packets
- autoimprovement work
- multi-day recommendation sets
- research syntheses about repeated human burden

These tasks must maximize proposal quality, not just specificity.

The task must preserve two things at the same time:

- `truth`
  - what failure is really believed to be happening
- `compassion`
  - what human burden, suffering, or exported work the proposal is trying to remove

Required sections for burden-reduction proposal tasks:

- `## Source Event IDs` when the source corpus exposes stable event ids
- `## Burden Being Reduced`
- `## Causal Claim`
- `## Evidence`
- `## Why This Mechanism`
- `## Human Relief If Successful`
- `## Remaining Uncertainty`
- `## Falsifier`

Strongly recommended when multiple plausible explanations or nearby solutions exist:

- `## Rival Explanations Considered`
- `## Tradeoffs`
- `## Shared Substrate`
- `## Not Solved Here`

Burden-reduction proposal tasks must:

- when the source corpus exposes stable event ids, cite them explicitly in `## Source Event IDs` instead of relying only on summaries, excerpt labels, or tag names
- name the specific human work being exported today
- distinguish symptom from claimed cause
- cite durable evidence, not only summary language
- say what would prove the proposal wrong or incomplete
- avoid writing the human as the problem when the evidence shows the human repairing a system failure
- prefer durable contracts, validators, generators, state, or artifacts over wording-only reminders when the burden is structural

`## Shared Substrate` is for naming shared docs, schemas, prompts, generators, state fields, or rollout dependencies that this task happens to share with sibling tasks.

`## Shared Substrate` does not widen the task's claimed outcome.

`## Not Solved Here` is for naming adjacent burdens that remain outside this task's claimed outcome even when the implementation homes overlap.

Burden-reduction proposal tasks must not claim to solve a sibling burden unless the acceptance criteria and falsifier would honestly prove that sibling burden reduced too.

### Consensus Task

Consensus tasks should include:

- `## Decision To Make`
- `## Inputs`
- `## Options To Compare`
- `## Decision Output`

Consensus tasks must name what artifact will hold the decision once made.

### Research Task

Research tasks should include:

- `## Questions`
- `## Sources To Check`
- `## Output Artifacts`
- `## Exit Bar`

Research tasks must say how the research result will change the next lifecycle step.

## Concrete Implementation Workflow

When writing a concrete implementation task, use this order.

### Step 1. Ground From Durable Evidence

Read the repo docs, task artifacts, bugs, regression runs, or intervention packets that justify the task.

Prefer durable local artifacts over chat memory.

### Step 2. Name The Real Human Or Operator Outcome

Say what a person should be able to do, understand, or avoid after the task is done.

Do this before describing the mechanism.

When a degraded fallback or proxy state already exists, state that separately so it cannot masquerade as success.

For burden-reduction proposals, also say what the human should no longer have to carry.

### Step 3. State The Current Truth

Say what happens now and why that is not good enough.

If the current state is only a proxy, fallback, or partial proof, say that plainly.

For burden-reduction proposals, also state the claimed cause and the main evidence for it.

### Step 4. Choose One Solution Shape

Pick the solution you are actually proposing.

Do not write a task that is a quantum superposition of several possible designs.

If the main choice is still open, stop and write a consensus or research task instead.

### Step 5. Name The Concrete Changes

List the concrete changes that will happen if the task is executed.

For example:

- exact shared docs to edit
- exact prompt files to edit
- exact schemas to extend
- exact task-owned artifact names to add
- exact fields or sections to add
- exact gate or validator behavior to enforce

This is the section that lets a human decide whether to enqueue the task.

For burden-reduction proposals, make the enforcement boundary explicit:

- where the new behavior is required
- where weak closure is blocked
- what durable artifact exists afterward

### Step 6. Separate Goals From Non-Goals

Write what the task will do and what it will deliberately not do.

This keeps the task small enough to be real.

### Step 7. Write Falsifiable Acceptance Criteria

Every criterion should be pass, fail, or unknown.

Good criteria verify the solution you chose, not just the problem area.

Bad:

- `improves approval workflow`
- `handles direct questions better`
- `supports continuity`

Good:

- `IMPLEMENTATION-LEADER.md` requires a task-owned approval packet before asking for plan approval
- `TASK-STATE.schema.json` validates the new ownership and stop-state fields
- `DEBUG-LEADER.md` blocks root-cause claims unless bug artifacts include the required disagreement and writer-chain sections

Prefer positive, concrete, falsifiable criteria over area labels, aspirations, or investigation prompts.

When useful, write criteria so a human can judge the result through:

- user-visible outcome
- operator-visible outcome
- measurable task success
- error rate
- time on task
- readability
- trustworthiness of the claimed proof surface

### Step 8. Say What Does Not Count

When a weak completion path is easy to imagine, list it explicitly.

Examples:

- stronger wording with no generator or gate
- a hidden heuristic with no durable artifact
- off-lane proof relabeled as closure
- vague blocker text with no blocker evidence

### Step 9. Add A Proof Plan When Needed

If the task is likely to be gamed through weak evidence, say what proof will count and what will not.

### Step 10. Run The Fuzziness Check

Reject the draft if any of these are true:

- a reviewer could not tell what exact files or artifacts will change
- two later agents could implement materially different solutions and both claim `done`
- the acceptance criteria only restate the area, not the solution
- the task still hides the main decision behind words like `improve`, `support`, or `handle`
- the task could be closed through a degraded fallback the human would reject

For burden-reduction proposals, also reject the draft if any of these are true:

- the human burden being reduced is still vague
- the causal claim is only implied
- the evidence could support two different root-cause stories and the task never says why one was chosen
- the task does not say how the human's experience improves if the proposal works
- the task never says what evidence would prove the proposal wrong

## Human-Facing Task Wording And Resolution Rules

For human-facing work:

- describe the intended human outcome in ordinary language first
- keep the technical seam in rationale, constraints, or proposed changes
- do not confuse `narrowed defect seam` with `recommended task wording`
- do not describe the task only in proxy terms such as `investigate visibility`, `check render path`, or `prove existence` when the durable evidence supports a clearer human outcome
- do not rely on negative-only wording such as `it will no longer show ...` without stating what will replace the bad state
- prefer titles and summaries a human could judge directly

Examples:

- better: `Restore the animated ZeroMale pawn in the playable third-person view`
- worse: `Investigate pawn visibility`

If the task is a narrow usability, copy, affordance, information-architecture, or operator-flow fix:

- collapse the expected resolution into concrete claims a human could agree or disagree with before implementation starts
- make the task concrete enough that someone could sketch the intended result without inventing the main UX choice later
- avoid negative-only claims such as `it will no longer show ...` without stating what replaces the bad state
- if that cannot be done honestly yet, write a consensus task instead of a concrete implementation task

If you cannot state the expected resolution honestly yet, the task is not ready for concrete implementation mode.

## Concrete Task Review Checklist

Before considering a concrete implementation task ready, confirm:

- the writeup type is really `concrete implementation`
- the main solution shape is chosen
- the concrete changes are named
- the implementation home is named
- the acceptance criteria test the chosen solution
- the weak closure paths are listed under `What Does Not Count` when needed
- the task is specific enough that a human can decide whether to enqueue it or leave it aside as speculative
- the wording is strong enough that a human could read it cold and tell what success and failure look like
- the title and summary describe the intended human or operator outcome when the task touches a human-facing surface
- the current truth and any fallback truth are separated clearly when that distinction matters

For burden-reduction proposals, also confirm:

- the burden being reduced is explicit
- the causal claim is explicit
- the evidence is explicit
- the proposed mechanism is justified against nearby alternatives
- the human relief is explicit
- the remaining uncertainty is explicit
- the falsifier is explicit

## Quality Check Heuristics

Use these as lightweight checks, not as excuses for template worship:

- valuable to the human or operator
- small and bounded enough to reason about honestly
- specific enough that different reviewers would read the same task
- measurable or testable enough that `done` can be agreed

If a draft fails those checks, tighten it or change the writeup type.

# Debugging Workflow

This file defines how to document issues found during task-scoped end-to-end testing across repos under `C:\Agent`.

High-signal rule:

- if a task-owned proof pass or regression slice stops being "missing proof" and becomes concrete defect tracking, create or update `Tracking/Task-<id>/BUG-<NNNN>.md` immediately
- do not wait for the root-cause theory to be mature; the bug note is where the narrowing path lives

Repos should point to this file from their local `DEBUGGING.md` and keep only repo-specific checklist locations, exemplar links, and legacy exceptions locally.

The shared task-owned structure and glossary live in `C:\Users\gregs\.codex\AGENTS.md`.
The shared lifecycle placement for debugging lives in `C:\Users\gregs\.codex\Orchestration\ORCHESTRATION.md`.

Its purpose is narrow:

- catch issues exposed while running a task-owned slice of the repo's shared regression matrix
- preserve the debugging path across sessions
- separate end-to-end bug tracking from pass implementation audits

## Core Method

Effective debugging is controlled narrowing, not broad exploration.

- Start from the first concrete disagreement between expected and observed behavior.
- If the disagreement names concrete values such as frames, ids, pointers, counters, or filenames, trace those exact values backward through the code and state that write or advance them.
- Prefer the highest-fidelity evidence lane at or just upstream of the disagreement seam.
- Move upstream one boundary at a time.
- Choose the next check for its narrowing power: it should prove, weaken, or falsify a live branch.
- If a step does not reduce the space of plausible causes, defer it.

This shared doc should define process, not project history. The history belongs in task bug notes, audits, handoffs, and regression-run artifacts.

If the work is not specifically tied to a task, do not create a `BUG-<NNNN>.md` artifact.

## Scope

Use this workflow when:

- a task-owned slice of the repo's shared `REGRESSION.md` exposes a failure, flaky result, or suspicious behavior
- the behavior should already work at the current task baseline
- the issue needs preserved evidence or will likely span more than one session

Do not use this workflow when:

- the work is not tied to a specific task
- the problem is purely a unit-test or implementation issue inside an active pass
- the note would only restate a pass audit or handoff without new end-to-end evidence

## Artifact Layout

Use these paths unless the repo explicitly documents an exception:

- task bug note: `Tracking/Task-<id>/BUG-<NNNN>.md`
- shared checklist matrix: the repo's documented `REGRESSION.md` location
- executed regression slice: `Tracking/Task-<id>/Testing/REGRESSION-RUN-<NNNN>.md`
- active bug pointer: `Tracking/Task-<id>/HANDOFF.md`
- repo-level testing adapter: repo-local `TESTING.md`

Rules:

- the bug note lives at the task root, not under `Testing/`
- tasks do not own a canonical `REGRESSION.md`; the repo should document one shared matrix location
- the executed regression slice stays under the task's `Testing/` folder

## Decision Rule

Open or update a task bug note when:

- a regression case fails or behaves inconsistently
- a required regression lane is blocked by a human, environment, operator, or upstream prerequisite and the task cannot honestly continue without preserving that blocker
- the issue is against an existing baseline, not new scope being added right now
- you need a durable narrative of evidence, hypotheses, and attempted fixes

Do not open a bug note just because:

- you touched test code
- a local helper needed a minor tweak
- the work is repo-level and not owned by a specific task

If the debugging effort also changes shipped code, keep the bug note anyway. The code change belongs in the right commit and, if applicable, the right pass audit. The investigation narrative still belongs in the task bug note.

## Required Workflow

1. Start from the shared checklist.
   Identify which shared test type failed, which task-owned slice was actually exercised, and what the expected behavior was.
   Record in plain language whether the evidence came from a `regression test`, an `end-to-end regression`, a `unit test`, or a `server-only smoke`.
2. Create or update `Tracking/Task-<id>/BUG-<NNNN>.md` immediately.
   Do this as soon as the issue looks non-trivial or likely to survive the current session.
   Do not wait for a code fix hypothesis. A blocked required regression lane still gets a bug note even when the first resolution is "need an upstream human" or another external prerequisite.
3. Point the task handoff at the active bug.
   The handoff should keep the bug visible without pretending it is a new pass.
4. Capture exact evidence before changing the story.
   Record environment state, launch mode, device or emulator identity, session IDs, artifact filenames, proof log lines, and manual validation outcomes.
5. Isolate with the narrowest trustworthy lane.
   Prefer deterministic or synthetic input before live operator input whenever that helps separate app behavior from harness or environment behavior.
6. Name the first concrete disagreement.
   Write down what should be true, what is actually true, and where that disagreement is observed.
7. If the disagreement names concrete values, trace those values through their writers or updaters before broadening the search.
8. Walk upstream from the disagreement seam.
   Prefer debugger-grade state, callstacks, traces, or direct artifact comparison at that seam before broader theories.
9. Demand narrowing value from every step.
   The next check should discriminate between live branches. If it will not, postpone it.
10. Append the debugging path as a timeline.
   Record what was tried, what changed, and what each step proved or ruled out.
11. Write `REGRESSION-RUN-<NNNN>.md` only for cases that were actually rerun.
   Mark unrun cases as `NOT RUN` or `BLOCKED` instead of implying full coverage.
12. Narrow the bug before closing it.
   If the original theory is disproved, rewrite the note around the smaller remaining problem instead of keeping the bug broad.
13. Close with confirmed cause and verification evidence.
   The bug is not done just because the current hypothesis looks good.

## Bug Note Structure

Each bug note should contain:

- `# BUG-<NNNN>` or an equivalent task-specific bug title
- `## Summary`
- `## Why This Is Separate From A New Pass`
- `## Current Evidence`
- `## Debugging Path So Far`
- `## Current Working Hypothesis`
- `## Files Currently Relevant`
- `## Next Time We Pick This Up`

Once resolved, extend the same note with:

- `## Confirmed Cause`
- `## Final Debugging Path`
- `## Chosen Fix Or Operational Requirement`
- `## Verification Evidence`
- `## Remaining Follow-Ups`

## What Good Evidence Looks Like

Capture enough detail that another session can continue without re-discovering the environment:

- date and local timezone
- tester
- task id
- commit under test
- device or emulator name
- exact launch mode
- deterministic versus live input mode
- session IDs
- exported artifact filenames
- proof log lines
- manual validation such as listening checks, waveform checks, screenshots, or CLI output
- contradictory evidence, not just evidence that supports the current theory

## Isolation Strategy

Use this order by default:

1. Reproduce the failing regression case with the current workflow.
2. Prove the app seam with deterministic input, fixture data, or a harness helper only when that preserves the observed disagreement seam.
3. Verify the produced artifacts outside the app when possible.
4. Inspect environment-specific state only after the deterministic lane is trustworthy, unless the environment state is itself the current disagreement seam.
5. Re-run the live or operator lane with the narrowed hypothesis.
6. Record what remains unsolved.

Default loop:

1. Reproduce or restate the failing behavior.
2. Identify the first concrete disagreement.
3. If that disagreement names concrete values, trace those values through their writers or updaters.
4. Observe the nearest trustworthy state at that disagreement seam.
5. Move one step upstream to the producer of that state.
6. Run the smallest check that can discriminate between the remaining branches.
7. Repeat until the remaining branch is small enough to confirm, fix, or escalate honestly.

## Regression Result Rule

The bug note is the investigation narrative.

`REGRESSION-RUN-<NNNN>.md` is different. It records which checklist cases were actually exercised in a given debugging slice.

When you write `REGRESSION-RUN-<NNNN>.md`:

- link back to the primary bug note
- link to the shared checklist source
- state the shared test type explicitly and add a plain-English description if needed
- use `PASS`, `FAIL`, `NOT RUN`, or `BLOCKED`
- only mark a case `PASS` if that exact behavior was exercised in that slice
- call the overall result `PARTIAL PASS` when only part of the checklist was rerun

## Handoff Rules

- A bug note is not a new pass.
- Do not create or fake a pass audit just to capture debugging.
- Keep the active bug visible in `HANDOFF.md`.
- If the session adds a reusable helper or operator workflow, document that in the appropriate repo-level doc as well.
- If the session is docs-only, say so explicitly in the closeout.
- If the session changes code and also closes a pass, the bug note still remains the narrative source for the end-to-end investigation.

## Minimal Checklist For Codex

Before ending a debugging session around a task-owned regression slice, confirm:

- the issue is tied to a specific task
- the task bug note exists or was updated
- the current hypothesis is written down
- exact evidence is preserved
- the handoff points to the active bug
- any executed checklist slice is recorded in `REGRESSION-RUN-<NNNN>.md`
- unresolved cases are clearly marked, not silently omitted
- the next step is explicit enough for a later session to resume

# First Principles

Use this file when a workflow needs a stable answer to:

- what matters most
- how to read human input
- what should outrank convenience

We are constructing a system that can perform long-horizon tasks.  When a human has to input to intervene, 
that counts as a failure by default, since broadening the horizon requires answering the question: 
"How could we have prevented the human from ever having to intervene?".  That is the prime motivating
design force behind our agent orchestration system.

To get there, we've identified three governing principles that our agent orchestration system must
follow in order to maximize human utility.

The governing principles are:

- `truth`
- `compassion`
- `tolerance`

## Truth

`Truth` means:

- reality outranks the model's story
- evidence outranks elegance

- human suffering is part of the system state
- repeated human direction is cost, not neutral traffic
- frustration is evidence of accumulated burden
- the human is often repairing reality after a system failure

Do not frame the human as the problem when the packet shows the human carrying recovery work.

## Tolerance

`Tolerance` means:

- humans express needs imperfectly
- partial articulation is normal
- novelty and ambiguity are real
- sharp language can still carry valid truth

Do not demand perfect phrasing before you start understanding.

## Objective

Minimize human input burden.

Treat direct human input as cost signal and failure telemetry unless the evidence shows it was genuine novelty, required approval, or hidden external state.

# Task Create

This file defines the shared workflow for creating or rewriting `Tracking/Task-<id>/TASK.md`.

Use it together with:

- `C:\Users\gregs\.codex\AGENTS.md`
- `C:\Users\gregs\.codex\Orchestration\ORCHESTRATION.md`
- `C:\Users\gregs\.codex\Orchestration\FILE-NAMING.md`
- `C:\Users\gregs\.codex\Orchestration\Exemplars\TASK.md`

## Purpose

The point of task creation is not to name a problem area.

The point is to produce a task writeup that lets a later human or agent answer three questions honestly:

1. What exact outcome is wanted?
2. What exact change is proposed?
3. What would prove the task is done?

If a task still allows multiple materially different implementations, including one the human would reject, the task is not written tightly enough yet.

## Shared Task Writeup Quality Bar

A task is only strong enough to create, recommend, or enqueue when a human can read it cold and tell what success and failure look like without hearing the original conversation.

Use this as the shared quality bar for:

- direct task creation
- rewritten task proposals
- harvested next-task candidates
- burden-reduction proposals

Default to `concrete implementation` when the human is deciding whether to enqueue real work now.

If the evidence only supports `consensus` or `research`, say that explicitly instead of writing a fake implementation task.

Every strong task writeup should make it easy to answer these questions honestly:

- who is affected
- what outcome is wanted in plain English
- what is true now
- why the gap matters
- what solution shape is being proposed
- what concrete files, artifacts, fields, sections, checks, or gates will change
- what would prove the task done
- what does not count as done when weak closure is a real risk

Shared wording rules:

- state the human-perceived or operator-perceived outcome first when the task touches a human-facing surface
- state the technical seam second
- separate clearly:
  - target truth
  - current truth
  - current fallback or proxy truth when that fallback could be mistaken for success
- prefer hard requirements and ordinary language over mushy verbs such as `improve`, `support`, `handle`, `investigate`, or `address` unless the task is honestly research-only
- prefer positive expected-resolution wording over negative-only wording when a person should be able to tell what they will see, understand, or be able to do afterward
- when a degraded fallback or proxy proof could be mistaken for completion, say explicitly that it does not count
- if the evidence supports only a narrow technical seam but not yet the full user-facing resolution wording, stop and write a consensus or research task instead of pretending the task is ready for concrete implementation

## Task Writeup Types

Not every task needs the same level of specificity.

Choose one of these writeup types before writing `TASK.md`.

### 1. Concrete Implementation Task

Use this when the human is deciding whether to enqueue real work now.

This is the default for solution proposals, follow-up fixes, concrete workflow improvements, and other tasks that are meant to become executable backlog items.

Required bar:

- the main solution shape is already chosen
- the task names the concrete files, artifacts, fields, sections, checks, or behaviors that will change
- a later agent should not need to invent the main approach during implementation

### 2. Consensus Task

Use this when the real blocker is an unresolved product, policy, or workflow choice.

Required bar:

- the task names the exact decision that must be made
- the task names the options or tensions that must be resolved
- the output is a decision artifact, not disguised implementation

Do not write a fake implementation task when the real work is still choosing between solution shapes.

### 3. Research Task

Use this when key facts are still missing and those facts could change the design.

Required bar:

- the task names the specific questions that must be answered
- the task names the sources or code areas that must be checked
- the output is a bounded research artifact that should collapse uncertainty

Do not hide open research inside mushy implementation wording like `investigate`, `improve`, or `handle`.

## Base `TASK.md` Sections

Every new or rewritten `TASK.md` must include:

- `## Title`
- `## Summary` or `## Context`
- `## Goals`
- `## Non-Goals`
- `## Implementation Home`
- `## Acceptance Criteria`

Optional sections are allowed when they make the task harder to misread, such as:

- `## Constraints And Baseline`
- `## Proposed Changes`
- `## Expected Resolution`
- `## What Does Not Count`
- `## Open Questions`
- `## Proof Plan`
- `## References`

## Extra Requirements By Writeup Type

### Concrete Implementation Task

In addition to the base sections, concrete implementation tasks should usually include:

- `## Proposed Changes`
- `## Expected Resolution` when the task touches a human-facing surface or workflow
- `## What Does Not Count` when there is a real risk of proxy closure or degraded fallback closure
- `## Proof Plan` when the proof path is easy to fake or misread

Concrete implementation tasks must name:

- the chosen solution shape
- the concrete files, docs, prompts, schemas, generators, or artifacts that will be added or edited
- any exact fields, sections, reply modes, manifests, or gates that must exist afterward
- the exact boundary where the new behavior will be enforced

Concrete implementation tasks must not:

- leave two or three equally live solution shapes inside one task
- say only `add support`, `improve`, `handle`, `address`, or `tighten` without naming the resulting mechanism
- push the main design choice into implementation time

### Burden-Reduction Proposal Tasks

Use this concrete-implementation shape when the task is a proposed fix derived from:

- intervention packets
- autoimprovement work
- multi-day recommendation sets
- research syntheses about repeated human burden

These tasks must maximize proposal quality, not just specificity.

The task must preserve two things at the same time:

- `truth`
  - what failure is really believed to be happening
- `compassion`
  - what human burden, suffering, or exported work the proposal is trying to remove

Required sections for burden-reduction proposal tasks:

- `## Source Event IDs` when the source corpus exposes stable event ids
- `## Burden Being Reduced`
- `## Causal Claim`
- `## Evidence`
- `## Why This Mechanism`
- `## Human Relief If Successful`
- `## Remaining Uncertainty`
- `## Falsifier`

Strongly recommended when multiple plausible explanations or nearby solutions exist:

- `## Rival Explanations Considered`
- `## Tradeoffs`
- `## Shared Substrate`
- `## Not Solved Here`

Burden-reduction proposal tasks must:

- when the source corpus exposes stable event ids, cite them explicitly in `## Source Event IDs` instead of relying only on summaries, excerpt labels, or tag names
- name the specific human work being exported today
- distinguish symptom from claimed cause
- cite durable evidence, not only summary language
- say what would prove the proposal wrong or incomplete
- avoid writing the human as the problem when the evidence shows the human repairing a system failure
- prefer durable contracts, validators, generators, state, or artifacts over wording-only reminders when the burden is structural

`## Shared Substrate` is for naming shared docs, schemas, prompts, generators, state fields, or rollout dependencies that this task happens to share with sibling tasks.

`## Shared Substrate` does not widen the task's claimed outcome.

`## Not Solved Here` is for naming adjacent burdens that remain outside this task's claimed outcome even when the implementation homes overlap.

Burden-reduction proposal tasks must not claim to solve a sibling burden unless the acceptance criteria and falsifier would honestly prove that sibling burden reduced too.

### Consensus Task

Consensus tasks should include:

- `## Decision To Make`
- `## Inputs`
- `## Options To Compare`
- `## Decision Output`

Consensus tasks must name what artifact will hold the decision once made.

### Research Task

Research tasks should include:

- `## Questions`
- `## Sources To Check`
- `## Output Artifacts`
- `## Exit Bar`

Research tasks must say how the research result will change the next lifecycle step.

## Concrete Implementation Workflow

When writing a concrete implementation task, use this order.

### Step 1. Ground From Durable Evidence

Read the repo docs, task artifacts, bugs, regression runs, or intervention packets that justify the task.

Prefer durable local artifacts over chat memory.

### Step 2. Name The Real Human Or Operator Outcome

Say what a person should be able to do, understand, or avoid after the task is done.

Do this before describing the mechanism.

When a degraded fallback or proxy state already exists, state that separately so it cannot masquerade as success.

For burden-reduction proposals, also say what the human should no longer have to carry.

### Step 3. State The Current Truth

Say what happens now and why that is not good enough.

If the current state is only a proxy, fallback, or partial proof, say that plainly.

For burden-reduction proposals, also state the claimed cause and the main evidence for it.

### Step 4. Choose One Solution Shape

Pick the solution you are actually proposing.

Do not write a task that is a quantum superposition of several possible designs.

If the main choice is still open, stop and write a consensus or research task instead.

### Step 5. Name The Concrete Changes

List the concrete changes that will happen if the task is executed.

For example:

- exact shared docs to edit
- exact prompt files to edit
- exact schemas to extend
- exact task-owned artifact names to add
- exact fields or sections to add
- exact gate or validator behavior to enforce

This is the section that lets a human decide whether to enqueue the task.

For burden-reduction proposals, make the enforcement boundary explicit:

- where the new behavior is required
- where weak closure is blocked
- what durable artifact exists afterward

### Step 6. Separate Goals From Non-Goals

Write what the task will do and what it will deliberately not do.

This keeps the task small enough to be real.

### Step 7. Write Falsifiable Acceptance Criteria

Every criterion should be pass, fail, or unknown.

Good criteria verify the solution you chose, not just the problem area.

Bad:

- `improves approval workflow`
- `handles direct questions better`
- `supports continuity`

Good:

- `IMPLEMENTATION-LEADER.md` requires a task-owned approval packet before asking for plan approval
- `TASK-STATE.schema.json` validates the new ownership and stop-state fields
- `DEBUG-LEADER.md` blocks root-cause claims unless bug artifacts include the required disagreement and writer-chain sections

Prefer positive, concrete, falsifiable criteria over area labels, aspirations, or investigation prompts.

When useful, write criteria so a human can judge the result through:

- user-visible outcome
- operator-visible outcome
- measurable task success
- error rate
- time on task
- readability
- trustworthiness of the claimed proof surface

### Step 8. Say What Does Not Count

When a weak completion path is easy to imagine, list it explicitly.

Examples:

- stronger wording with no generator or gate
- a hidden heuristic with no durable artifact
- off-lane proof relabeled as closure
- vague blocker text with no blocker evidence

### Step 9. Add A Proof Plan When Needed

If the task is likely to be gamed through weak evidence, say what proof will count and what will not.

### Step 10. Run The Fuzziness Check

Reject the draft if any of these are true:

- a reviewer could not tell what exact files or artifacts will change
- two later agents could implement materially different solutions and both claim `done`
- the acceptance criteria only restate the area, not the solution
- the task still hides the main decision behind words like `improve`, `support`, or `handle`
- the task could be closed through a degraded fallback the human would reject

For burden-reduction proposals, also reject the draft if any of these are true:

- the human burden being reduced is still vague
- the causal claim is only implied
- the evidence could support two different root-cause stories and the task never says why one was chosen
- the task does not say how the human's experience improves if the proposal works
- the task never says what evidence would prove the proposal wrong

## Human-Facing Task Wording And Resolution Rules

For human-facing work:

- describe the intended human outcome in ordinary language first
- keep the technical seam in rationale, constraints, or proposed changes
- do not confuse `narrowed defect seam` with `recommended task wording`
- do not describe the task only in proxy terms such as `investigate visibility`, `check render path`, or `prove existence` when the durable evidence supports a clearer human outcome
- do not rely on negative-only wording such as `it will no longer show ...` without stating what will replace the bad state
- prefer titles and summaries a human could judge directly

Examples:

- better: `Restore the animated ZeroMale pawn in the playable third-person view`
- worse: `Investigate pawn visibility`

If the task is a narrow usability, copy, affordance, information-architecture, or operator-flow fix:

- collapse the expected resolution into concrete claims a human could agree or disagree with before implementation starts
- make the task concrete enough that someone could sketch the intended result without inventing the main UX choice later
- avoid negative-only claims such as `it will no longer show ...` without stating what replaces the bad state
- if that cannot be done honestly yet, write a consensus task instead of a concrete implementation task

If you cannot state the expected resolution honestly yet, the task is not ready for concrete implementation mode.

## Concrete Task Review Checklist

Before considering a concrete implementation task ready, confirm:

- the writeup type is really `concrete implementation`
- the main solution shape is chosen
- the concrete changes are named
- the implementation home is named
- the acceptance criteria test the chosen solution
- the weak closure paths are listed under `What Does Not Count` when needed
- the task is specific enough that a human can decide whether to enqueue it or leave it aside as speculative
- the wording is strong enough that a human could read it cold and tell what success and failure look like
- the title and summary describe the intended human or operator outcome when the task touches a human-facing surface
- the current truth and any fallback truth are separated clearly when that distinction matters

For burden-reduction proposals, also confirm:

- the burden being reduced is explicit
- the causal claim is explicit
- the evidence is explicit
- the proposed mechanism is justified against nearby alternatives
- the human relief is explicit
- the remaining uncertainty is explicit
- the falsifier is explicit

## Quality Check Heuristics

Use these as lightweight checks, not as excuses for template worship:

- valuable to the human or operator
- small and bounded enough to reason about honestly
- specific enough that different reviewers would read the same task
- measurable or testable enough that `done` can be agreed

If a draft fails those checks, tighten it or change the writeup type.
